## 4. 단계 사이 상태 전달 방식 ★

**방식: HttpSession 누적 + (OCR만) 중간 DB 저장**(옛과 동일). 프론트 일괄 제출이나 product_join_progress 중간 DB 저장은 **사용하지 않음**.

| 데이터 | 저장 위치 | 저장 시점 | 근거(ref) |
|---|---|---|---|
| 약관 동의 코드 | HttpSession `PRODUCT_JOIN_TERMS` | 단계3 saveTermsToSession | `ProductJoinServiceImpl.java:172` |
| **우대금리 쿠폰(선택)** | HttpSession `PRODUCT_JOIN_COUPON` | **단계3.5 saveCouponToSession** | `ProductJoinServiceImpl.java:370` (신규) |
| 가입 폼(금액/기간/통화/출금계좌/계좌비번) | HttpSession `PRODUCT_JOIN_FORM` | 단계6 saveJoinFormToSession | `ProductJoinServiceImpl.java:337` |
| OCR 검증번호(verificationNo) | HttpSession `PRODUCT_JOIN_VERIFICATION_NO` + **id_verifications 테이블** | 단계5a(별도 @Transactional) | 세션 `:220`,`:266` / INSERT `:218`,`:264` |
| 휴대폰 인증 통과 플래그 | HttpSession `PRODUCT_JOIN_PHONE_VERIFIED_PRODUCT_NO`(컨트롤러 setAttribute) + **인증코드는 Redis `PHONE_VERIFY:{phone}`(3분 TTL)** | 단계5b | 세션 `ProductJoinController.java:232` / Redis `SolapiSmsService.java:78` |

**"지금 입력 데이터가 어디 존재하는가"(확정 전 시점, 항목별)**:
- 약관 동의 내용, 가입 폼 입력값, **쿠폰 선택** → **오직 HttpSession에만 존재**(톰캣 인메모리 추정). DB에는 없음.
- OCR 본인인증 → **id_verifications 테이블에 이미 INSERT됨** + 세션 verificationNo(가입 트랜잭션과 분리된 선행 커밋).
- 휴대폰 본인인증 → **세션 플래그만**(DB 레코드 없음). 인증코드 자체는 **Redis에 임시 보관**되며 verifyCode 성공 시 즉시 삭제(`SolapiSmsService.java:88`) → 인증 완료 기록 미영속.
- 전자서명 → 확정 전 어디에도 없음. 클라 canvas `signatureImageData`가 complete 요청 body에 담겨 전송될 때 처음 등장.

**completeJoin이 세션에서 꺼내 검증/사용**: `getAttribute(SESSION_TERMS/SESSION_FORM/SESSION_COUPON/SESSION_VERIFICATION_NO)` — `ProductJoinServiceImpl.java:396-406`(**COUPON 읽기 신규**).
**완료 후 정리**: `clearJoinSession`이 TERMS/FORM/VERIFICATION_NO/**COUPON 4개** removeAttribute — `ProductJoinServiceImpl.java:718-723`. **`PRODUCT_JOIN_PHONE_VERIFIED_PRODUCT_NO`는 여전히 제거하지 않아 가입 완료 후 세션에 잔존**(옛 gap 그대로).

**불명확/주의**: 세션이 톰캣 인메모리(기본)면 서버 재시작/스케일아웃 시 진행 중 데이터 소실 가능(spring-session 미연동, §1). 설정 부재로 인한 추정.
