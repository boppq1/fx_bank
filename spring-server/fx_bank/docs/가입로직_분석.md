> 경로 표기: 일부 에이전트는 `src/...` 상대경로로 보고했으나, 실제 루트는 `fx_bank/spring-server/fx_bank/src/...`이다(동일 파일).

**프론트 방식**: 하이브리드. `@Controller`(ProductJoinPageController)가 `templates/product/join/**` SSR 뷰를 반환하고, 각 페이지의 inline `<script>`가 `fetch()`로 `@RestController`(`/api/product/join/**`)를 호출한다. 상품 가입 전용 static JS 파일은 없고 모든 JS는 템플릿 inline이다. fetch는 Authorization Bearer 토큰 + `credentials:'same-origin'`(세션쿠키 동반) 사용 (`form.html:124-161`, `ProductController.java:23-24`).

**HttpSession 사용 여부**: 사용함 (★ 상품 가입 흐름 한정). 서블릿 표준 HttpSession에 약관/폼/OCR검증번호/휴대폰인증 상태를 단계별로 보관. 로그인/사용자 식별은 세션이 아니라 JWT + Redis(`USER:`+userId)로 처리.
- 세션 키 4개: `PRODUCT_JOIN_TERMS` / `PRODUCT_JOIN_FORM` / `PRODUCT_JOIN_VERIFICATION_NO` / `PRODUCT_JOIN_PHONE_VERIFIED_PRODUCT_NO` (`ProductJoinServiceImpl.java:56-59`)
- **불일치(주의)**: `SecurityConfig.java:32-33`은 "세션 안 씀" 주석과 함께 STATELESS 선언하나, 실제 join 로직은 HttpSession에 의존. STATELESS는 SecurityContext를 세션에 저장하지 않는다는 의미일 뿐 서블릿 HttpSession 생성 자체는 막지 않음.
- spring-session / `@EnableRedisHttpSession` / `spring.session.store-type` 설정 **없음** → HttpSession은 기본 톰캣 인메모리 세션으로 동작 추정(Grep 0건). Redis는 토큰·사용자정보 전용.

**가입 패키지 구조** (`com.example.bank.product`):
- controller (4): `ProductController`, `ProductJoinController`(REST), `ProductJoinPageController`(뷰), `ProductReviewController`
- service: `ProductJoinService`(IF) / `ProductJoinServiceImpl`, `ProductService`(IF) / `ProductServiceImpl`
- dao: `IProductDao`, `IProductJoinDao` (MyBatis 매퍼 IF)
- dto: 27개 (가입 관련 14개 포함)
- mapper XML: `mapper/mybatis/productJoin.xml`(가입 전용), `product.xml`(상품 메타)
- templates: `product/join/{terms, terms-reader, form, complete}.html`

## 4. 단계 사이 상태 전달 방식 ★

**방식: HttpSession 누적 + (OCR만) 중간 DB 저장**. 프론트 일괄 제출이나 product_join_progress 중간 DB 저장은 **사용하지 않음**.

| 데이터 | 저장 위치 | 저장 시점 | 근거(ref) |
|---|---|---|---|
| 약관 동의 코드 | HttpSession `PRODUCT_JOIN_TERMS` | 단계3 saveTermsToSession | `ProductJoinServiceImpl.java:169` |
| 가입 폼(금액/기간/통화/출금계좌/계좌비번) | HttpSession `PRODUCT_JOIN_FORM` | 단계6 saveJoinFormToSession | `ProductJoinServiceImpl.java:334` |
| OCR 검증번호(verificationNo) | HttpSession `PRODUCT_JOIN_VERIFICATION_NO` + **id_verifications 테이블** | 단계5a (별도 @Transactional) | `ProductJoinServiceImpl.java:217`, `:263` / INSERT `:215`, `:261` |
| 휴대폰 인증 통과 플래그 | HttpSession `PRODUCT_JOIN_PHONE_VERIFIED_PRODUCT_NO` (컨트롤러가 setAttribute) | 단계5b | `ProductJoinController.java:202` |

**"지금 입력 데이터가 어디 존재하는가" (확정 전 시점)**:
- 약관 동의 내용, 가입 폼 입력값 → **오직 HttpSession에만 존재**(톰캣 인메모리 추정). DB에는 없음.
- OCR 본인인증 → **id_verifications 테이블에 이미 INSERT됨** + 세션에 verificationNo 보관 (가입 트랜잭션과 분리된 선행 커밋).
- 휴대폰 본인인증 → **세션 플래그만 존재**, DB 레코드 없음.
- 전자서명 → 확정 전에는 어디에도 없음. 클라이언트의 `signatureImageData`로 complete 요청 body에 담겨 전송될 때 처음 등장 (`ProductJoinSubmitRequestDto`).

**completeJoin이 세션에서 꺼내 검증/사용**: `getAttribute(SESSION_TERMS/SESSION_FORM/SESSION_VERIFICATION_NO)` — `ProductJoinServiceImpl.java:368-375`.
**완료 후 정리**: `clearJoinSession`이 TERMS/FORM/VERIFICATION_NO 3개만 removeAttribute — `ProductJoinServiceImpl.java:644-648`. **`PRODUCT_JOIN_PHONE_VERIFIED_PRODUCT_NO`는 제거하지 않아 가입 완료 후에도 세션에 잔존**(gap).

**불명확/주의**: 세션이 톰캣 인메모리(기본)면 서버 재시작/스케일아웃 시 진행 중 데이터 소실 가능(spring-session 미연동, §1). 단정은 불가(설정 부재로 인한 추정).
